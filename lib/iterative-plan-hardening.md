# Plan Hardening — закалка плана перед кодированием

> N проходов ревью с чистым контекстом каждый раз. План становится чище, баги умирают до написания кода.

---

## Принцип

```
План v0 → Ревью (свежий контекст!) → План v1 → Ревью → ... → План vN (закалённый) → КОДИРОВАНИЕ
```

**Ключевое**: каждый проход — СВЕЖИЙ контекст. Нет "слепых пятен" от привыкания к тексту.

## Почему работает

1. **Разные "глаза"** — чистый контекст фокусируется на разных аспектах каждый раз
2. **Баг в плане дешевле бага в коде** — shift-left до предела, 0 cost fix
3. **Diminishing returns = сигнал остановки** — только стилистика → план готов
4. **Дёшево** — 3-5 проходов по плану vs 100x дороже по коду

---

## Реализация в tero2

Convergence loop — автоматическая закалка до стабилизации:

```python
while True:
    result = review_agent(plan, "найди неточности, баги, противоречия")
    if result.issues_found == 0 or result.only_cosmetic:
        break  # план закалён
    plan = apply_fixes(plan, result)
    if (iteration := iteration + 1) >= MAX_HARDENING_ROUNDS:  # 3-5
        break
```

Reviewer в режиме Plan Review (`system-prompts.md`). Каждая итерация — новый агент (Iron Rule).

## Связь с другими паттернами

| Паттерн | Связь |
|---|---|
| **Adversarial Self-Review** | То же для КОДА. Plan Hardening = shift-left на этап плана |
| **Coach review** | Coach ревьюит ПОСЛЕ. Plan Hardening — ДО выполнения |
| **SRepair CoT** | "Думай на NL перед кодом" — тот же принцип на уровне фикса |
| **Мультипатч** | Несколько кандидатов для кода. Здесь — несколько проходов для ПЛАНА |

## Метрика успеха

**Plan Bug Escape Rate** — % багов в коде, предсказуемых из плана. Цель: в коде остаются только непредвидимые баги.
