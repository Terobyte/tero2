# GSD-2 — паттерны для tero2

> Ключевые архитектурные идеи из GSD-2, адаптированные под tero2.
> Полная реализация: [detailed-roadmap.md](detailed-roadmap.md)

---

## Железное Правило (The Iron Rule)

> Одна задача = одно контекстное окно. Если не помещается — декомпозируй.

- Каждая задача стартует с **чистого контекста** (до 200K токенов)
- Нет накопления мусора: ошибок, неверных попыток, устаревшего кода
- Качество на 1000-й строке = качество на 1-й строке

**Pre-execution проверка**: 80% окна → warning, 95% → hard fail.
**Target compaction**: 70% использования контекста.

---

## Иерархия работы

```
Milestone (веха)      — крупный этап, 4-10 срезов
  └── Slice (срез)     — вертикальная фича, 1-7 задач
       └── Task (задача) — атомарная единица, ≤ 1 контекстное окно
```

---

## Disk Layer — Spec-Driven Development (SDD)

Диск = абсолютный источник истины. Состояние экстернализировано из памяти LLM в файловую систему.

### Структура `.sora/` (аналог `.gsd/`)

| Категория | Файлы |
|-----------|-------|
| Макро-состояние | `PROJECT.md`, `REQUIREMENTS.md`, `DECISIONS.md` |
| Межсессионная память | `KNOWLEDGE.md`, `RUNTIME.md` |
| Текущий статус | `STATE.md` |
| Per-milestone | `milestones/M001/` — контекст, план, риски |
| Per-slice | `slices/S01/` — план, summary, UAT |

**Цикл SDD**: спецификация → код → тесты → review → корректировка спецификации.

---

## Context Assembly (предзагрузка контекста)

Каждая задача стартует с чистого окна. В промпт динамически инжектируются:
- План текущей задачи и среза
- Резюме выполненных задач (lossy compression)
- Зависимости и выжимки
- Дорожная карта и реестр решений (lossless compression)

**Результат**: агент получает отфильтрованную суть, без информационного мусора.

---

## Машина состояний (Auto-Mode)

Каждый срез проходит строгую цепочку фаз:

```
Plan → Execute → Complete → Reassess Roadmap → Validate Milestone
```

| Фаза | Что делает |
|------|-----------|
| **Plan** | Сканирует кодовую базу, анализирует зависимости, декомпозирует на задачи |
| **Execute** | Задача в изолированном окне + auto-fix retries (линтеры, тесты) |
| **Complete** | Резюме → SUMMARY.md, UAT-скрипт, коммит |
| **Reassess** | Circuit breaker: имеет ли остаток roadmap смысл? |
| **Validate** | Сверка критериев успеха с фактическими результатами |

---

## Stuck Detection

Контекстная форензика для выявления петель:
- Детекция бесконечных циклов исправлений
- Эскалация: доп. ресурс → прерывание с диагностикой
- Стратегии: inject / defer / replan / quick-task

---

## Динамическая маршрутизация моделей

**Классификатор сложности** — детерминированный, sub-ms:
- Анализ AST, структуры директорий, размера файлов, типа задачи
- Без сетевых запросов к LLM

| Сложность | Модель | Когда |
|-----------|--------|-------|
| simple | Haiku ($0.25/M) | Рефакторинг, документация, синтаксис |
| standard | Sonnet ($3/M) | Типовые фичи |
| complex | Opus ($15/M) | Архитектура, планирование, глубокие баги |

### Budget Pressure Guards

- Пороги: 50% → 75% → 90% от лимита → автоматический tier shifting
- Профили: `budget` → `balanced` (default) → `quality`
- Реестр затрат в реальном времени

---

## Параллелизм (Worktree Isolation)

Несколько агентов работают одновременно над независимыми вехами.

**Принцип**: каждый воркер = отдельная ветка + физическая директория (Git Worktrees).
Нет race conditions: изолированные файлы, тесты, env vars.

**Ключевые концепции**:
- Диспетчеризация с учётом графа зависимостей
- Проверка пересечений редактируемых файлов
- Детекция конфликтов в реальном времени
- Squash-слияние с гарантией целостности

---

## Отказоустойчивость

### Трёхуровневые тайм-ауты

| Уровень | Время | Действие |
|---------|-------|---------|
| Soft | 8 мин | Steering: "please wrap up" |
| Idle | 3 мин без активности | "Are you stuck?" |
| Hard | 15 мин | Принудительная остановка + сохранение стейта |

### Восстановление

- 429 rate-limit → exponential backoff + bounded retry budget
- Crash recovery: восстановление с последнего подтверждённого шага
- Single-Writer State Engine: эксклюзивная блокировка файлов (OS-level)

---

## Главные идеи для tero2

1. **Железное правило**: 1 задача = 1 контекстное окно (решает Context Rot)
2. **Диск как источник истины**: экстернализация стейта из LLM в файловую систему
3. **Context Assembly**: динамическая инжекция только нужного контекста
4. **Классификатор сложности**: sub-ms маршрутизация задач → подходящая модель
5. **Worktree-параллелизм**: несколько агентов без race conditions
6. **Stuck Detection**: автоматическое обнаружение и выход из петель
7. **Трёхуровневые тайм-ауты**: soft → idle → hard
8. **Budget Guards**: автоматический tier shifting при приближении к лимиту
