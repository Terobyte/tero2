# Gemma-local провайдер + Telegram-чат с «рычагами» — идея (не spec)

**Статус:** идея, записана для будущей детализации. Не реализуется в рамках текущего TUI UX Redesign.
**Дата:** 2026-04-17
**Источник:** диалог с пользователем во время брейнсторма TUI redesign.

---

## Суть идеи

Добавить в tero2 **локальный Gemma-провайдер**, работающий через общий протокол, и использовать его как голос Telegram-чата — чтобы пользователь мог говорить с tero2 через мессенджер, а не только отправлять планы.

## Ключевые архитектурные решения пользователя

1. **Gemma = локальный провайдер tero2.** Не зависит от OpenVerb; код копируется в tero2 и работает автономно. Контракт — общий HTTP-протокол (llama.cpp server или аналог).

2. **НЕТ tool calling.** Gemma не вызывает функции сама. Вместо этого у Telegram-чата есть **предзаготовленный набор «рычагов»** — команд с чётко прописанными именами и описаниями «что делать и как». Gemma в ответе выбирает один из рычагов (или никакой — просто текстовый ответ). Парсер на стороне tero2 распознаёт «рычаг» в ответе и применяет его.

   > Причина: tool calling на маленьких моделях (~2B) ненадёжен. Явный каталог «рычагов» — предсказуемее и проще отладить.

3. **Fallback по команде пользователя:** если Gemma «тупит», пользователь **явно** говорит ей переключиться → Gemma передаёт эстафету Claude Sonnet через OpenRouter API (ключ есть у пользователя). Никакой автоматической эскалации — только по явной команде.

## Что это НЕ

- НЕ замена Scout/Builder/Architect — это отдельная роль «говорящий ассистент».
- НЕ двусторонний chat с самим агентом-runner'ом — Gemma говорит *о* runner'е, но не *внутри* него.
- НЕ зависимость от OpenVerb — OpenVerb используется только как источник проверенного кода работы с Gemma; протокол общий.

## Открытые вопросы для будущего spec'а

- **Каталог «рычагов»:** какие команды? Минимум: `status`, `pause`, `resume`, `skip_task`, `switch_provider(role, cli, model)`, `show_last_error`, `escalate_to_sonnet`. Каждая с именем, параметрами, описанием.
- **Формат ответа Gemma:** JSON-блок `{action: "pause"}` или structured text? Парсер должен быть толерантен к мусору вокруг.
- **Escalation к Sonnet:** отдельный провайдер `openrouter`? Или расширение существующего `claude` провайдера с новым endpoint'ом?
- **Где живёт Gemma-процесс:** tero2 сам запускает llama-server как subprocess, или ожидает запущенный на `localhost:8080`?
- **Модельный файл:** хранится в `~/.tero2/models/gemma-e2b.gguf`? Скачивается автоматически при первом запуске?
- **Telegram-UX:** превращается ли текущий `TelegramInputBot` ([tero2/telegram_input.py](../../tero2/telegram_input.py)) в chat-bot, или это новый отдельный сервис рядом?
- **Голос:** принимаем ли голосовые сообщения (Gemma audio-native это умеет)? Это убирает необходимость в отдельном STT.

## Связь с текущим TUI-redesign

В текущем spec'е оставляется **место в каталоге провайдеров** для `gemma` (6-й провайдер после `zai/claude/codex/opencode/kilo`). UI должен уметь показать Gemma в ModelPickScreen без правок, когда провайдер появится. Сам провайдер — отдельным spec'ом.
