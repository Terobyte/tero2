# MVP2.5 "Стратег" — Full SORA Cycle

> **Status:** Design approved
> **Depends on:** MVP0 (Immortal Runner), MVP1 (Reflexion + Telegram), MVP2 (Stuck Detection + Escalation)
> **Goal:** Transform the single-executor runner into a multi-role SORA pipeline: Scout → Architect → Coach → Builder → Verifier with Plan Hardening, Persona system, and Context Assembly.

---

## 1. Problem

MVP0-2 implemented crash recovery, reflexion, stuck detection, and escalation — but the runner still has a single "executor" role that receives the entire plan and runs it blindly. The core SORA multi-role pipeline (originally planned for MVP1-2) was deferred. Without it:

- No codebase reconnaissance before building (Scout)
- No automatic task decomposition (Architect)
- No strategic re-evaluation mid-run (Coach)
- No quality gate separating code writing from verification (Builder vs Verifier)
- No plan quality improvement before execution (Plan Hardening)
- No role-specific prompt optimization (Persona + Context Assembly)

## 2. Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Runner structure | Phase Handlers (option B) | Runner is already ~380 lines; monolith would bloat it. Phases test independently. |
| Role config | Explicit config required | No hardcoded defaults. User chooses providers per role in config.toml. Future CLI will manage this. |
| SORA activation | Presence of `[roles.builder]` | No builder in config → legacy executor mode. Zero breaking changes. |
| Naming | MVP2.5 | Preserves MVP3-7 roadmap numbering. Signals this is deferred MVP1-2 work. |
| Scope | Full SORA | All deferred roles: Scout, Architect, Coach, Builder, Verifier + Plan Hardening + Persona + Context Assembly + Triggers. |

## 3. State Model Extensions

`state.py` gains SORA phase tracking:

```python
class SoraPhase(str, Enum):
    NONE = "none"              # legacy executor mode (no SORA)
    HARDENING = "hardening"    # Plan Hardening
    SCOUT = "scout"
    COACH = "coach"
    ARCHITECT = "architect"
    EXECUTE = "execute"        # Builder → Verifier loop
    SLICE_DONE = "slice_done"  # between slices, Coach re-evaluation
```

**AgentState new fields** (add to `AgentState` dataclass in `state.py`):
```python
# AgentState gains these three fields:
sora_phase: SoraPhase = SoraPhase.NONE
current_slice: str = ""         # e.g. "S01"
current_task_index: int = 0     # 0-based index within slice
```

**Serialization:** Both `to_json()` and `from_json()` must handle `SoraPhase` explicitly, same as `Phase`:
```python
# to_json() — add alongside existing d["phase"] = self.phase.value:
d["sora_phase"] = self.sora_phase.value

# from_json() — add alongside existing Phase coercion:
d["sora_phase"] = SoraPhase(d.get("sora_phase", "none"))
```
Old STATE.json files without `sora_phase` default to `SoraPhase.NONE` — backwards compatible.

**Crash recovery:** Runner reads `sora_phase` from STATE.json and jumps to the corresponding phase handler. `sora_phase == NONE` → fallback to legacy executor mode. `sora_phase == SLICE_DONE` → re-run coach phase (idempotent), then continue to next slice from TASK_QUEUE.

**Config changes between runs:** If a role is removed from config after a crash (e.g. `[roles.reviewer]` removed while `sora_phase=HARDENING`), the runner skips the now-absent phase silently and advances to the next available phase. `sora_phase` moves forward when the next phase calls `set_sora_phase`. No manual state reset is required. If the removed role was required (e.g. builder/architect/verifier), `load_config()` raises `ConfigError` before the runner starts.

**Phase transitions:** Runner updates `sora_phase` via `CheckpointManager.set_sora_phase(state, phase)` *before* launching each phase. This new method updates `state.sora_phase`, writes to disk, and returns the updated state. Crash between phases = re-run the same phase (each handler is idempotent).

## 4. Persona System

New module: `tero2/persona.py`

```python
@dataclass
class Persona:
    name: str           # "scout", "builder", etc.
    system_prompt: str   # .md file content (body after frontmatter)
    metadata: dict       # from YAML frontmatter (role, model_hint, max_tokens)

class PersonaRegistry:
    def __init__(self, disk: DiskLayer) -> None: ...
    def load(self, role_name: str) -> Persona: ...
    def load_or_default(self, role_name: str) -> Persona: ...
```

**Prompt file format** (`.sora/prompts/{role}.md`):
```markdown
---
role: scout
model_hint: cheap
max_tokens: 4000
---
You are Scout, a fast codebase reconnaissance agent...
```

**Frontmatter parsing:** Simple regex parser (no PyYAML). Delimiter `---`, key: value per line.

**Caching:** In-memory for duration of run. No persistence — re-read on restart (file may have changed).

**Bundled prompts:** MVP2.5 ships default prompts for all roles as package data in `tero2/prompts/` (included via `pyproject.toml` `[tool.setuptools.package-data]`). User can override by placing their own in `.sora/prompts/`.

**Loading priority:** `PersonaRegistry.load(role)` checks `.sora/prompts/{role}.md` first (project-level override), then falls back to `importlib.resources` to read `tero2/prompts/{role}.md` (bundled default). If neither exists → `FileNotFoundError`.

## 5. Context Assembly

New module: `tero2/context_assembly.py`

```python
class BudgetState(str, Enum):
    OK = "ok"               # estimated_tokens < config.context.target_ratio * window
    WARNING = "warning"     # target_ratio ≤ estimated < warning_ratio
    COMPRESS = "compress"   # warning_ratio ≤ estimated < hard_fail_ratio — start cutting
    HARD_FAIL = "hard_fail" # estimated ≥ hard_fail_ratio — refuse to run, raise error

@dataclass
class AssembledPrompt:
    system_prompt: str      # persona prompt (never cut)
    user_prompt: str        # assembled content
    budget_state: BudgetState
    estimated_tokens: int

class ContextAssembler:
    def __init__(self, disk: DiskLayer, config: Config, personas: PersonaRegistry): ...
    def assemble_scout(self) -> AssembledPrompt: ...
    def assemble_architect(self, slice_id: str) -> AssembledPrompt: ...
    def assemble_builder(self, task_plan: str, reflexion_prompt: str, context_hints: str) -> AssembledPrompt: ...
    def assemble_verifier(self, task_plan: str) -> AssembledPrompt: ...
    def assemble_coach(self) -> AssembledPrompt: ...
    def assemble_reviewer(self, plan: str, mode: str) -> AssembledPrompt: ...
```

**Role-specific assembly:**

| Role | Content | Target window |
|------|---------|--------------|
| Scout | persona + PROJECT.md + file tree | Minimal |
| Architect | persona + STRATEGY + CONTEXT_MAP + ROADMAP | 15-25K tokens |
| Builder | persona + task plan + context hints + reflexion_prompt (caller calls `ReflexionContext.to_prompt()`) | 8-15K tokens |
| Verifier | persona + task plan + must-haves | Minimal |
| Coach | persona + ROADMAP + CONTEXT_MAP + summaries + EVENT_JOURNAL + metrics | 30-50K tokens |
| Reviewer | persona + plan + mode (find-issues / apply-fixes) | 15-25K tokens |

**Truncation priority (cut first → last):**
1. Old SUMMARYs (oldest first)
2. CONTEXT_MAP
3. Code snippets
4. CONTEXT_HINTS
5. Task plan — **never**
6. System prompt — **never**

**Token estimation:** Heuristic `len(text) / 4`. Accurate to ~10% for English/code.

## 6. Players

All players live in `tero2/players/` with a common interface.

### 6.1 Common Interface

```python
# tero2/players/base.py

@dataclass
class PlayerResult:
    success: bool
    output_file: str       # what was written to disk (relative to .sora/)
    captured_output: str   # text for reflexion/logging
    error: str = ""

async def run(
    chain: ProviderChain,
    prompt: AssembledPrompt,
    checkpoint: CheckpointManager,
    state: AgentState,
    config: Config,
) -> PlayerResult
```

### 6.2 Scout (`players/scout.py`)

- **Input:** PROJECT.md + file tree listing
- **Output:** `milestones/M001/CONTEXT_MAP.md` — compressed code map (key files, architecture, dependencies)
- **Failure:** Non-fatal. Architect works without map (reduced quality).
- **Skip condition:** Project < `skip_scout_if_files_lt` files (configurable, default 20)

### 6.3 Architect (`players/architect.py`)

- **Input:** STRATEGY + CONTEXT_MAP + PLAN.md (hardened plan if available, otherwise ROADMAP.md as fallback)
  - Fallback logic: `disk.read_file("milestones/M001/PLAN.md")` → if empty → `disk.read_file("milestones/M001/ROADMAP.md")`
- **Output:** `milestones/M001/S01/S01-PLAN.md` — always S01 for the first (and currently only) slice. Multi-slice iteration is driven by TASK_QUEUE.md from Coach; Architect is called once per slice with `slice_id` passed in.
  - Slice numbering: `architect_phase.py` receives `slice_id: str` (e.g. `"S01"`) from the runner's TASK_QUEUE loop. It calls `run_architect(ctx, slice_id)` and writes to `milestones/M001/{slice_id}/{slice_id}-PLAN.md`.
  - `run_architect` signature: `async def run_architect(ctx: RunnerContext, slice_id: str = "S01") -> PhaseResult`

**Player vs phase handler:** `players/architect.py` is the `Architect` player class — handles LLM interaction (prompt assembly via `ContextAssembler`, `ctx.run_agent()` call, `_parse_slice_plan()` output parsing). `phases/architect_phase.py` is the `run_architect()` phase orchestrator — calls `Architect.run()`, handles the result, writes the plan file, returns `PhaseResult`. The phase handler wraps the player; each has a separate file.

- **Failure:** Fatal for Slice — escalation

**S0X-PLAN.md format** (written by Architect, parsed by `_parse_slice_plan()`):

```markdown
## T01: Task name
Task description — what needs to be built or changed.

**Must-haves:**
- condition 1 (verifiable, specific)
- condition 2

## T02: Task name
Task description.

**Must-haves:**
- condition 1
```

Rules the Architect prompt enforces:
- Each task starts with `## T0X:` (two `#`, zero-padded index, colon, space, name)
- `**Must-haves:**` section is required for every task (may be `- none` if truly no conditions)
- No sub-sections, no other `##` headers inside task blocks

**Parser** (`_parse_slice_plan(content: str, slice_id: str) -> SlicePlan`):
- Split on `^## T\d+:` regex
- For each block: extract task id from header, description from body before `**Must-haves:**`, must_haves from bullet list after it
- If parsing fails (malformed LLM output) → `PhaseResult(success=False, error="malformed slice plan — cannot parse tasks")`  → fatal, Architect is re-invoked (counts as Architect failure)

### 6.4 Builder (`players/builder.py`)

- **Input:** task plan + context hints + reflexion context
- **Output:** code in project directory + `T0X-SUMMARY.md`
- **Failure:** reflexion → retry (existing mechanism)
- **Reuses:** `ctx.run_agent()` from RunnerContext — stuck detection + step tracking work as-is

### 6.5 Verifier (`players/verifier.py`)

- **Required in SORA mode.** `load_config()` raises `ConfigError` if `roles.builder` is present but `roles.verifier` is absent (see §9 config validation).
- **Input:** task plan + must-haves
- **Runs:** `ruff check .` + `pytest -x` (if configured) — `verifier.py` executes these as **subprocesses directly** (not via LLM). It reads their stdout/stderr, counts errors, and synthesizes the verdict string. The LLM Verifier persona is used only when tests pass but semantic must-haves need checking (conditions that cannot be verified by ruff/pytest alone).
- **Returns:** `PlayerResult` where `captured_output` starts with `"PASS"`, `"FAIL"`, or `"ANOMALY"` on the first line. `execute_phase` checks `result.captured_output.splitlines()[0].strip()` to determine verdict. **If `captured_output` is empty, treat as `ANOMALY`.**
- **FAIL →** reflexion → Builder retry
- **ANOMALY →** Verifier writes to `persistent/EVENT_JOURNAL.md`, `execute_phase` then calls `check_triggers()` and fires Coach if needed

**Verifier verdict format:**
```
PASS
All must-haves satisfied. ruff: 0 errors. pytest: 3 passed.
```
```
FAIL
must-have "tokens expire after 1h" not satisfied
pytest: test_token_expiry FAILED
```
```
ANOMALY
Unexpected behaviour: database connection refused during test
Possible environment issue — not a code bug.
```

### 6.6 Coach (`tero2/players/coach.py`)

- **Input:** Everything from disk (ROADMAP, CONTEXT_MAP, summaries, EVENT_JOURNAL, metrics)
- **Output:** STRATEGY.md, TASK_QUEUE.md, RISK.md, CONTEXT_HINTS.md
- **Iron Rule:** Each invocation = fresh context (no accumulation)
- **Invocation:** FIRST_RUN (before Architect) and END_OF_SLICE (after Execute) are always-on fixed pipeline steps — not conditional events. ANOMALY, STUCK, BUDGET_60, and HUMAN_STEER are reactive triggers fired by other phase handlers. Despite using the `CoachTrigger` enum, FIRST_RUN and END_OF_SLICE are sequential and guaranteed; the others are conditional.
- **Failure:** Non-fatal. Runner continues with last STRATEGY.md (or without one).
- **Returns:** `PlayerResult` like all other players. `output_file` = `"strategic/STRATEGY.md"`.

### 6.7 Plan Hardening (`tero2/plan_hardening.py`)

- **Convergence loop:** max 5 rounds
- **Round =** Reviewer(find-issues) → if critical > 0 → Reviewer(apply-fixes)
- **Intermediate versions:** `milestones/M001/plan_v1.md`, `milestones/M001/plan_v2.md`, ...
- **Final output:** `milestones/M001/PLAN.md` (replaces ROADMAP.md as the execution source)
- **No separate Reviewer player.** Plan Hardening calls `ContextAssembler.assemble_reviewer()` and invokes the provider chain directly. The Reviewer is a persona, not a player class.
- **Stop condition:** critical issues == 0 OR max rounds reached

**Critical vs cosmetic issues:**
Reviewer prompt instructs the model to output a structured block:
```
CRITICAL: <description>   ← missing steps, impossible dependencies, missing tech stack info
COSMETIC: <description>   ← wording, formatting, minor clarifications
NO ISSUES FOUND           ← explicit marker when plan is complete and correct
```
`plan_hardening.py` stop conditions:
- `NO ISSUES FOUND` present in Reviewer output → stop (plan is good)
- Zero `CRITICAL:` lines AND `stop_on_cosmetic_only=True` → stop (only cosmetic remain)
- Max rounds reached → stop regardless

**Fallback for malformed output:** If Reviewer output contains neither `CRITICAL:`, `COSMETIC:`, nor `NO ISSUES FOUND` (e.g. prose response), treat as a single critical issue: `CRITICAL: reviewer output malformed — re-run`. This prevents silently accepting a plan when the Reviewer failed to follow format. After 2 consecutive malformed responses, log a warning and treat as `NO ISSUES FOUND` (don't loop forever on a broken reviewer).

### 6.8 Standalone Hardening Mode

Plan Hardening can run independently — without Scout, Architect, Builder, or Verifier. This enables iterating on a plan before committing to execution.

**CLI command:**
```
tero2 harden <project_path> --plan <plan.md> [--rounds N] [--debug]
```

- `--rounds N` — override `max_rounds` from config (default 5). Useful for quick checks (`--rounds 1`) or deep polishing (`--rounds 10`).
- `--debug` — verbose output: prints each round's full Reviewer output, critical/cosmetic counts, and the diff between plan versions.

**How it works:**

```python
# cli.py
def cmd_harden(args) -> None:
    """Run Plan Hardening standalone — review + fix loop without full SORA."""
    project_path = Path(args.project)
    plan_file = Path(args.plan)
    config = load_config(project_path)
    if args.rounds:
        config.plan_hardening.max_rounds = args.rounds

    disk = DiskLayer(project_path)
    disk.init()

    # Copy plan to milestones/M001/ROADMAP.md if not already there
    milestone = disk.sora_dir / "milestones" / "M001"
    milestone.mkdir(parents=True, exist_ok=True)
    roadmap = milestone / "ROADMAP.md"
    if not roadmap.exists() or plan_file != roadmap:
        roadmap.write_text(plan_file.read_text())

    # Build minimal RunnerContext (no builder/verifier/scout needed)
    ctx = _build_harden_context(project_path, config, disk, debug=args.debug)
    result = asyncio.run(run_harden(ctx))

    if result.success:
        print(f"✓ Plan hardened: {milestone / 'PLAN.md'}")
    else:
        print(f"✗ Hardening failed: {result.error}")
```

**Minimal RunnerContext for hardening:** Only requires `[roles.reviewer]` in config. No builder, scout, architect, or verifier needed. `_build_harden_context()` creates a RunnerContext with only the reviewer chain available.

**Debug output per round:**
```
── Round 1/5 ──────────────────────────────────
Reviewer found:
  CRITICAL: Step 3 depends on step 5 (impossible ordering)
  CRITICAL: No test strategy specified
  COSMETIC: Typo in step 2 description
Critical: 2, Cosmetic: 1 → applying fixes...
Saved: milestones/M001/plan_v1.md

── Round 2/5 ──────────────────────────────────
Reviewer found:
  COSMETIC: Step 2 could be clearer
Critical: 0, Cosmetic: 1 → stopping (no critical issues)
Final: milestones/M001/PLAN.md
```

**`run_harden()` debug support:** `harden_phase.py`'s `run_harden(ctx)` checks `ctx.config.plan_hardening.debug` (new bool field) to enable verbose printing. Each round logs: round number, full Reviewer output, critical/cosmetic count, action taken (fix or stop), and saved version path.

**Config addition:**
```python
@dataclass
class PlanHardeningConfig:
    max_rounds: int = 5
    stop_on_cosmetic_only: bool = True
    debug: bool = False              # ← new: verbose per-round output
```

**Use cases:**
- Quick plan check before long SORA run: `tero2 harden . --plan plan.md --rounds 2`
- Deep plan polishing with full trace: `tero2 harden . --plan plan.md --rounds 10 --debug`
- CI integration: run hardening as a pre-check, fail if critical issues remain

## 7. Phase Handlers

New directory: `tero2/phases/`

### 7.1 RunnerContext

```python
# tero2/phases/context.py

@dataclass
class Task:
    index: int           # 0-based within slice, matches current_task_index
    id: str              # e.g. "T01", "T02"
    description: str     # full task text from S0X-PLAN.md
    must_haves: list[str]  # extracted bullet list of must-have conditions

@dataclass
class SlicePlan:
    slice_id: str        # e.g. "S01"
    slice_dir: str       # relative to .sora/, e.g. "milestones/M001/S01"
    tasks: list[Task]    # ordered list of tasks to execute

@dataclass
class PhaseResult:
    success: bool
    error: str = ""
    data: dict = field(default_factory=dict)  # phase-specific data, e.g. {"slice_plan": SlicePlan}

@dataclass
class RunnerContext:
    project_path: Path
    config: Config
    disk: DiskLayer
    checkpoint: CheckpointManager
    notifier: Notifier
    personas: PersonaRegistry
    assembler: ContextAssembler
    cb_registry: CircuitBreakerRegistry
    state: AgentState
    shutdown_event: asyncio.Event   # for clean SIGTERM/SIGINT handling in SORA mode
    # Escalation state (moved from Runner instance to RunnerContext for SORA mode)
    escalation_level: EscalationLevel = EscalationLevel.NONE
    div_steps: int = 0
    escalation_history: list[EscalationLevel] = field(default_factory=list)
    # Event bus (MVP2.6 — None = headless mode, no events emitted)
    dispatcher: "EventDispatcher | None" = None
    command_queue: "asyncio.Queue[Command] | None" = None
    # Implementation note: guard imports with TYPE_CHECKING to avoid ImportError
    # when MVP2.5 is implemented before MVP2.6:
    #   from __future__ import annotations
    #   from typing import TYPE_CHECKING
    #   if TYPE_CHECKING:
    #       from tero2.events import EventDispatcher, Command
    milestone: str = "M001"

    def build_chain(self, role_name: str) -> ProviderChain:
        """Build provider chain for a given role from config.roles[role_name].

        Same logic as existing Runner._build_chain() but parameterized by role_name
        instead of hardcoded "executor". Reads provider + fallback + model from
        config.roles[role_name], creates providers via create_provider(), wraps in
        ProviderChain with CB registry.
        """

    async def run_agent(
        self,
        chain: ProviderChain,
        prompt_text: str,
    ) -> tuple[bool, str]:
        """Run provider chain, return (success, captured_output).

        Extracted from Runner._run_agent() into RunnerContext so Builder player
        can reuse stuck detection + step tracking without a Runner reference.
        Reads timeout from config.roles[role_name].timeout_s (caller resolves role).
        Uses self.checkpoint.increment_step() and self.state for step/tool tracking.

        Internally spawns a heartbeat coroutine (via self.notifier) that sends
        periodic 'still working' Telegram messages every
        config.telegram.heartbeat_interval_s seconds. The heartbeat task is
        cancelled before run_agent() returns (in a finally block).
        """
```

`run_agent()` is the key shared primitive. Builder player calls `ctx.run_agent(chain, prompt_text)` — no Runner reference needed, no code duplication.

### 7.2 Phase Handlers

Each phase is a module in `tero2/phases/`:

| File | Function | What it does |
|------|----------|-------------|
| `harden_phase.py` | `run_harden(ctx) → PhaseResult` | Plan Hardening convergence loop |
| `scout_phase.py` | `run_scout(ctx) → PhaseResult` | Scout → CONTEXT_MAP.md |
| `coach_phase.py` | `run_coach(ctx, trigger) → PhaseResult` | Coach → STRATEGY.md etc. |
| `architect_phase.py` | `run_architect(ctx, slice_id="S01") → PhaseResult` | Architect → S0X-PLAN.md |
| `execute_phase.py` | `run_execute(ctx, slice_plan) → PhaseResult` | Builder→Verifier loop per Task |

**`execute_phase.py` — ReflexionContext lifecycle:**

`run_execute` iterates tasks from `slice_plan`. Before iterating, reset `ctx.state.current_task_index = 0` when starting a new slice (caller sets this before calling `run_execute` for any slice after S01 — see §7.3 slice loop). For each task:
1. Update `ctx.state.current_task_index = task.index` and checkpoint it **before** starting Builder. This records which task is in progress so crash recovery re-enters the correct task.
2. Create a fresh `ReflexionContext()` per task (reflexion resets between tasks, not between slices).
   - Before entering the inner retry loop, load initial `context_hints` from disk: `context_hints = ctx.disk.read_file("strategic/CONTEXT_HINTS.md")` (empty string if absent). This is refreshed inside the retry loop when STEER.md is updated (step 4 below), but must be initialized here so the first iteration has it available.
3. **Interrupted task detection:** if recovering from crash (`task.index == ctx.state.current_task_index` on re-entry and previous attempt was not completed), seed reflexion with an interrupted-task note:
   ```python
   if recovering and task.index == resume_from_index:
       reflexion_ctx = add_attempt(
           reflexion_ctx,
           builder_output="(task was interrupted mid-execution — partial file changes may exist)",
           verifier_feedback="interrupted — review existing state before continuing",
       )
   ```
4. Inner retry loop (up to `config.reflexion.max_cycles + 1` attempts):
   - Check `ctx.shutdown_event.is_set()` — return early if set
   - Check `human/OVERRIDE.md` — if STOP directive → abort; if PAUSE directive → wait until cleared
   - Check `human/STEER.md` — if updated since last check → reload context_hints
   - Call `ctx.assembler.assemble_builder(task_plan, reflexion_ctx.to_prompt(), context_hints)`
   - Call `ctx.run_agent(chain, assembled.user_prompt)`
   - **Stuck/escalation:** if Builder output triggers stuck detection, call `decide_escalation()` / `execute_escalation()` from `escalation.py` (same logic as legacy runner). `action.should_pause` → return `PhaseResult(success=False, error="escalated to STUCK")` — caller fires `CoachTrigger.STUCK`.
   - On Builder failure: `reflexion_ctx = add_attempt(reflexion_ctx, builder_output=captured_output, verifier_feedback="attempt failed — no verifier output")`
   - On Verifier FAIL: parse verifier output for failed tests and unmet must-haves, then: `reflexion_ctx = add_attempt(reflexion_ctx, builder_output=captured_output, verifier_feedback=verifier_output, failed_tests=parsed_failures, must_haves_failed=parsed_must_haves)`
   - Note: `add_attempt` mutates `reflexion_ctx` in place AND returns it — always reassign: `reflexion_ctx = add_attempt(...)`

Crash recovery within execute phase: `ctx.state.current_task_index` tells `run_execute` which task to resume at. `resume_from_index` = `ctx.state.current_task_index` on entry; tasks before it are skipped. The resumed task gets the interrupted-task reflexion seed (step 3 above).

### 7.3 Runner Restructure

**Jump-to-phase crash recovery:** `_execute_sora` checks `state.sora_phase` at entry and skips already-completed phases:

```python
# Phase completion order (used to skip phases on crash recovery)
_PHASE_ORDER = [
    SoraPhase.HARDENING,
    SoraPhase.SCOUT,
    SoraPhase.COACH,
    SoraPhase.ARCHITECT,
    SoraPhase.EXECUTE,
    SoraPhase.SLICE_DONE,
]

def _phase_already_done(current: SoraPhase, candidate: SoraPhase) -> bool:
    """Return True if candidate phase precedes current phase (already completed)."""
    try:
        return _PHASE_ORDER.index(candidate) < _PHASE_ORDER.index(current)
    except ValueError:
        return False  # NONE or unknown — don't skip
```

Each `if "role" in config.roles:` block is guarded by `if not _phase_already_done(state.sora_phase, SoraPhase.X):`.

```python
# runner.py

async def _execute_sora(self, state: AgentState, shutdown_event: asyncio.Event) -> None:
    ctx = self._build_runner_context(state, shutdown_event)

    # Copy plan file → ROADMAP.md so Architect's fallback path always works.
    # Done once at entry (idempotent — overwrite is safe, plan file hasn't changed).
    roadmap_path = f"milestones/{ctx.milestone}/ROADMAP.md"
    if self.plan_file:
        ctx.disk.write_file(roadmap_path, self.plan_file.read_text())

    # 1. Plan Hardening (optional — requires [roles.reviewer])
    if "reviewer" in self.config.roles and not _phase_already_done(state.sora_phase, SoraPhase.HARDENING):
        state = self.checkpoint.set_sora_phase(state, SoraPhase.HARDENING)
        await run_harden(ctx)

    # 2. Scout (optional — requires [roles.scout])
    if "scout" in self.config.roles and not _phase_already_done(state.sora_phase, SoraPhase.SCOUT):
        state = self.checkpoint.set_sora_phase(state, SoraPhase.SCOUT)
        await run_scout(ctx)

    # 3. Coach first run (optional — requires [roles.coach])
    if "coach" in self.config.roles and not _phase_already_done(state.sora_phase, SoraPhase.COACH):
        state = self.checkpoint.set_sora_phase(state, SoraPhase.COACH)
        await run_coach(ctx, CoachTrigger.FIRST_RUN)

    # 4. Architect (required)
    if not _phase_already_done(state.sora_phase, SoraPhase.ARCHITECT):
        state = self.checkpoint.set_sora_phase(state, SoraPhase.ARCHITECT)
        result = await run_architect(ctx, slice_id="S01")
        if not result.success:
            await ctx.notifier.notify(f"Architect failed: {result.error}", NotifyLevel.ERROR)
            return  # fatal for this slice — abort SORA run
        slice_plan = result.data["slice_plan"]  # SlicePlan object
    else:
        # Recovering from crash mid-execute: re-parse slice plan from disk
        slice_plan = _load_slice_plan_from_disk(ctx, state.current_slice or "S01")

    # 5. Execute: Builder → Verifier per task (resumes from state.current_task_index on crash)
    if not _phase_already_done(state.sora_phase, SoraPhase.EXECUTE):
        state = self.checkpoint.set_sora_phase(state, SoraPhase.EXECUTE)
    await run_execute(ctx, slice_plan)

    # 6. Slice loop — complete, re-evaluate, fetch next slice, repeat
    # END_OF_SLICE and subsequent Architect/Execute calls are NOT guarded by
    # _phase_already_done — the while loop is only entered after initial Execute
    # completes. Crash recovery inside the loop re-enters via sora_phase
    # (SLICE_DONE, ARCHITECT, or EXECUTE) set at the top of each iteration.
    max_slices = self.config.max_slices  # guard against infinite Coach generation
    extra_slices_done = 0  # counts slices beyond initial S01 (total = extra_slices_done + 1)
    while extra_slices_done < max_slices:
        state = self.checkpoint.set_sora_phase(state, SoraPhase.SLICE_DONE)
        if "coach" in self.config.roles:
            await run_coach(ctx, CoachTrigger.END_OF_SLICE)

        next_slice = _read_next_slice(ctx)  # parses TASK_QUEUE.md, returns next unchecked ID or None
        if next_slice is None:
            break  # TASK_QUEUE exhausted — plan complete

        state.current_slice = next_slice
        state.current_task_index = 0   # reset index for new slice — crash recovery uses this
        state = self.checkpoint.save(state)
        extra_slices_done += 1

        state = self.checkpoint.set_sora_phase(state, SoraPhase.ARCHITECT)
        result = await run_architect(ctx, slice_id=next_slice)
        if not result.success:
            await ctx.notifier.notify(f"Architect failed on {next_slice}: {result.error}", NotifyLevel.ERROR)
            break
        slice_plan = result.data["slice_plan"]

        state = self.checkpoint.set_sora_phase(state, SoraPhase.EXECUTE)
        await run_execute(ctx, slice_plan)
    else:
        await ctx.notifier.notify(
            f"extra slice limit reached ({max_slices} additional slices beyond S01) — stopping. Check TASK_QUEUE.md.",
            NotifyLevel.ERROR,
        )

async def _execute_plan(self, state, shutdown_event):
    if "builder" in self.config.roles:
        await self._execute_sora(state, shutdown_event)
    else:
        await self._execute_legacy(state, shutdown_event)
```

**`_read_next_slice(ctx) → str | None`:** Reads `strategic/TASK_QUEUE.md` via `ctx.disk.read_file()`. Finds the first unchecked line matching `- [ ] SXX:`, marks it in-progress (`- [~] SXX:`), writes back, and returns the slice ID (e.g. `"S02"`). Returns `None` if no unchecked slices remain. Lives in `phases/context.py`.

**`_load_slice_plan_from_disk(ctx, slice_id) → SlicePlan`:** Reads `milestones/M001/{slice_id}/{slice_id}-PLAN.md` and parses it into a `SlicePlan` object. Used only on crash recovery when `sora_phase` is already past ARCHITECT. Lives in `phases/architect_phase.py` (same module that writes the file).

**Crash recovery for multi-slice:** `state.current_slice` is checkpointed before each Architect call. On re-entry, the recovery branch uses `state.current_slice or "S01"` — not hardcoded `"S01"`.

**Phase handlers check `ctx.shutdown_event.is_set()`** at natural boundaries (between tasks, between hardening rounds, etc.) and return early with `PhaseResult(success=False, error="shutdown requested")` if set.

**Legacy mode:** Current `_execute_plan()` code moves to `_execute_legacy()`. Zero breaking changes when no roles configured beyond executor.

**`run_agent()` refactoring — legacy compatibility:** `RunnerContext.run_agent()` is the canonical implementation (replaces `Runner._run_agent()`). `_execute_legacy()` creates a minimal `RunnerContext` via `self._build_runner_context(state, shutdown_event)` — the same builder used by SORA mode. This gives legacy mode access to `run_agent()` without code duplication. Legacy `_execute_legacy()` holds a reference to `ctx` and calls `ctx.run_agent(chain, prompt)` instead of `self._run_agent(chain, prompt)`. `Runner._run_agent()` is deleted.

## 8. Triggers

New module: `tero2/triggers.py`

```python
class CoachTrigger(str, Enum):
    FIRST_RUN = "first_run"
    END_OF_SLICE = "end_of_slice"
    ANOMALY = "anomaly"
    BUDGET_60 = "budget_60"
    STUCK = "stuck"
    HUMAN_STEER = "human_steer"

@dataclass
class TriggerResult:
    should_fire: bool
    trigger: CoachTrigger | None
    reason: str = ""

def check_triggers(state: AgentState, disk: DiskLayer, config: Config) -> TriggerResult:
    """Check all conditions, return highest priority trigger."""
```

**Priority (conditional triggers only):** STUCK > ANOMALY > HUMAN_STEER > BUDGET_60

`FIRST_RUN` and `END_OF_SLICE` are **not conditional triggers** — they are fixed pipeline steps called directly by the runner (always-on). They are not checked via `check_triggers()` and do not appear in the priority list.

**Where triggers are checked:**
- `END_OF_SLICE` — runner calls `run_coach(ctx, CoachTrigger.END_OF_SLICE)` directly at top of slice loop (not via `check_triggers`)
- `FIRST_RUN` — runner calls `run_coach(ctx, CoachTrigger.FIRST_RUN)` directly whenever `[roles.coach]` is configured (not conditional on STRATEGY.md existence, not via `check_triggers`)
- `ANOMALY` — execute_phase after Verifier returns ANOMALY, via `check_triggers()`
- `STUCK` — execute_phase when escalation reaches Level 2, via `check_triggers()`
- `HUMAN_STEER` — checked in two places: (1) inside execute_phase task loop when `human/STEER.md` is updated (triggers context_hints reload only); (2) by the runner at phase boundaries via `check_triggers()`, which may fire `run_coach(ctx, CoachTrigger.HUMAN_STEER)`. The task-loop check is a local reload; the phase-boundary check is what triggers Coach.
- `BUDGET_60` — runner at phase boundaries (reads metrics.json), via `check_triggers()`

**Coach failure is non-blocking.** If Coach fails, runner logs warning and continues without fresh strategy.

**Coach timeout:** Coach uses `config.roles["coach"].timeout_s` (same RoleConfig field as other roles). Default: `DEFAULT_PROVIDER_TIMEOUT_S` (same constant as executor). If timeout → `PhaseResult(success=False, error="coach timeout")` — runner continues.

## 10. Disk Data Flow

| Writer | File | Reader |
|--------|------|--------|
| User (manual) | `persistent/PROJECT.md` | Scout. Describes project tech stack, constraints, and goals. Created by the user before running SORA — `project_init` does NOT create this file. Scout degrades gracefully if absent (uses file tree only, logs a warning). |
| User (manual) | `.sora/prompts/{role}.md` | PersonaRegistry (project-level prompt override — optional; falls back to `tero2/prompts/{role}.md` if absent) |
| User / Telegram | `milestones/M001/ROADMAP.md` | Plan Hardening (raw plan input) |
| Plan Hardening | `milestones/M001/PLAN.md` | Architect (hardened plan; fallback to ROADMAP.md if no hardening) |
| Scout | `milestones/M001/CONTEXT_MAP.md` | Architect, Coach, Context Assembly |
| Coach | `strategic/STRATEGY.md` | Architect, Context Assembly |
| Coach | `strategic/TASK_QUEUE.md` | Runner (next Slice) |
| Coach | `strategic/CONTEXT_HINTS.md` | Context Assembly → Builder |
| Coach | `strategic/RISK.md` | (informational) |
| Architect | `milestones/M001/S01/S01-PLAN.md` | Runner (iterates Tasks) |
| Builder | `milestones/M001/S01/T0X-SUMMARY.md` | Context Assembly, Coach |
| execute_phase | `reports/metrics.json` | Coach (via `assemble_coach()`). Written after each task: `{task_id, slice_id, success, attempts, steps, duration_s}`. Uses `ctx.disk.write_file("reports/metrics.json", json.dumps(metrics))` — no new `disk_layer.py` method needed. |
| Verifier | `persistent/EVENT_JOURNAL.md` | Triggers → Coach |
| Human | `human/STEER.md` | execute_phase task loop (reload context_hints) + Triggers → Coach (HUMAN_STEER) |
| Human | `human/OVERRIDE.md` | execute_phase task loop (STOP/PAUSE directives checked each iteration) |

**TASK_QUEUE.md format** (written by Coach, parsed by Runner):
```markdown
## Slices

- [~] S01: Authentication system (in progress)
- [ ] S02: API endpoints
- [x] S03: Database migrations (completed)

## Priority notes
S01 before S02 — API depends on auth.
```
Runner reads checkboxes: `- [ ]` = pending, `- [~]` = in progress, `- [x]` = done. `_read_next_slice()` takes first `[ ]` entry, marks it `[~]`, writes back. Simple markdown, human-editable.

**Error handling:** If TASK_QUEUE.md is missing, empty, or has no unchecked slices → Runner treats the plan as complete and logs a warning. No infinite loop.
