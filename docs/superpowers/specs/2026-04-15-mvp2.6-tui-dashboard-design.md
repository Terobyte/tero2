# MVP2.6 "Dashboard" — TUI + Remote Control

> **Status:** Design approved
> **Depends on:** MVP0 (Immortal Runner), MVP1 (Reflexion + Telegram), MVP2 (Stuck Detection + Escalation), MVP2.5 (Full SORA Cycle)
> **Goal:** Full-screen terminal dashboard (Textual) with live process visualization, usage tracking, hot-swap providers, and interactive stuck resolution — plus remote control via Telegram with the same capabilities.

---

## 1. Problem

MVP0-2.5 built a powerful autonomous runner with SORA pipeline, reflexion, escalation, and Telegram notifications. But there is no way to SEE what's happening or CONTROL it in real time:

- No live visualization of pipeline phases, steps, retries
- No way to change providers mid-run without editing config files
- No usage/quota visibility — you don't know when a provider is about to hit its limit
- When stuck, the runner writes files and sends Telegram text — but there's no interactive dialog
- No persistent "always-on" mode — runner exits after plan completion

## 2. Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| TUI framework | Textual | Async-native (matches Runner), CSS-like styling, built-in widgets (Log, DataTable, ProgressBar), hotkeys. Best Python TUI framework. |
| Runner ↔ TUI coupling | Event Bus (Approach A) | Runner stays independent. TUI/Telegram/nothing can attach. Runner works headless on servers. |
| Communication | asyncio.Queue (events down, commands up) | Zero external deps. Works in-process. Same mechanism for TUI and Telegram. |
| Usage: account limits | External CLI (`caut`) | Supports 16+ providers, handles cookies/OAuth. Building this ourselves = massive scope. |
| Usage: session tracking | Built-in (parse provider output) | We already capture output for reflexion. Adding token counting is trivial. |
| Layout | Panel-based (like htop/lazygit) | User chose this. One screen, all info visible, no tab switching. |

## 3. Event Bus

Runner and consumers (TUI, Telegram) communicate through a two-way "radio" system.

### 3.1 Events (Runner → consumers)

```python
# tero2/events.py

@dataclass
class Event:
    timestamp: datetime
    kind: str       # event type
    role: str       # "builder", "scout", etc. or "" for system events
    data: dict      # event-specific payload

# Event kinds:
#   "phase_change"   — SORA phase transition (data: phase, slice, task)
#   "step"           — step within a task (data: step, max, action)
#   "stuck"          — runner is stuck (data: signal, level, options)
#   "done"           — plan completed (data: tasks_completed, time_elapsed)
#   "error"          — non-recoverable error (data: message)
#   "provider_switch"— provider changed (data: role, old, new)
#   "usage_update"   — usage data refreshed (data: limits, session_tokens, session_cost)
#   "escalation"     — escalation level changed (data: level, action)
#   "log"            — general log message (data: message, severity)
```

### 3.2 Commands (consumers → Runner)

```python
@dataclass
class Command:
    kind: str       # command type
    data: dict      # command-specific payload

# Command kinds:
#   "switch_provider" — data: {"role": "builder", "provider": "claude"}
#   "skip_task"       — data: {} (skip current task, move to next)
#   "steer"           — data: {"text": "try using async instead"}
#   "pause"           — data: {}
#   "resume"          — data: {}
#   "stop"            — data: {}
#   "retry"           — data: {} (retry current task from scratch)
#   "new_plan"        — data: {"text": "plan in docs/plan.md"} (free text — Runner tries
#                       as file path first, falls back to Coach interpretation if not a valid path)
```

### 3.3 EventDispatcher

A single `asyncio.Queue` only delivers to one consumer (`.get()` removes the item). TUI and Telegram both need every event. Solution: `EventDispatcher` — Runner pushes events to it, dispatcher copies to all subscribers.

```python
# tero2/events.py

class EventDispatcher:
    """Fan-out dispatcher. Runner calls emit(), consumers subscribe().

    Each subscriber gets its own asyncio.Queue. Events are copied to all.
    """

    def __init__(self) -> None:
        self._subscribers: list[asyncio.Queue[Event]] = []

    def subscribe(self) -> asyncio.Queue[Event]:
        """Create and return a new queue for a consumer."""
        q: asyncio.Queue[Event] = asyncio.Queue()
        self._subscribers.append(q)
        return q

    def unsubscribe(self, q: asyncio.Queue[Event]) -> None:
        self._subscribers.remove(q)

    async def emit(self, event: Event) -> None:
        """Copy event to all subscriber queues."""
        for q in self._subscribers:
            await q.put(event)
```

**Setup:**
```python
dispatcher = EventDispatcher()
command_queue: asyncio.Queue[Command] = asyncio.Queue()

# TUI subscribes
tui_events = dispatcher.subscribe()

# Telegram subscribes
tg_events = dispatcher.subscribe()

# Runner emits — both consumers get the event
await dispatcher.emit(Event(...))
```

Runner receives `dispatcher: EventDispatcher | None` and `command_queue: asyncio.Queue[Command] | None` at init. If dispatcher is None — headless mode, no events emitted. Runner checks `command_queue` at phase boundaries (between tasks, between slices, after stuck detection). Commands from TUI and Telegram go into the same `command_queue`.

**Command ordering:** Commands are processed FIFO. Stale commands (e.g., "resume" when not paused) are silently ignored — Runner checks preconditions before acting.

## 5. Hot-Swap Providers

### 5.1 TUI Flow

1. Press `[r]` → modal popup with all roles and current providers
2. Select a role → second popup with available providers (claude, codex, opencode, kilo)
3. Select provider → popup closes, TUI sends `Command(kind="switch_provider", data={"role": "builder", "provider": "claude"})`
4. Runner receives command at next boundary → updates `config.roles[role].provider` in memory → next step uses new provider
5. Pipeline panel immediately reflects the change

```
┌─ Roles ──────────────────────────┐
│                                   │
│  scout:     kilo     [изменить]  │
│  architect: claude   [изменить]  │
│  builder:   opencode [изменить]  │
│  verifier:  codex    [изменить]  │
│  coach:     claude   [изменить]  │
│                                   │
│  [Esc] закрыть                   │
└───────────────────────────────────┘
```

### 5.2 Telegram Flow

Same capability, text-based. Commands use `/` prefix to distinguish from plan text:
- `/builder claude` → switches builder to claude
- `/scout kilo` → switches scout to kilo
- `/status` → current roles + phases + usage
- `/skip` → skip current task
- `/pause` → pause runner
- `/steer try async instead` → inject steer text

Messages without `/` prefix are treated as plan text (existing behavior).

## 6. Usage Tracking

### 6.1 Account Limits (via `caut`)

New module: `tero2/usage_tracker.py`

```python
class UsageTracker:
    async def fetch_limits(self) -> dict[str, float]:
        """Call `caut usage --json`, return {provider: percent_remaining}.

        Expected caut output (JSON array of provider objects):
        [{"provider": "claude", "usage_pct": 22.0, "remaining_pct": 78.0, ...}, ...]

        We extract `remaining_pct` per provider. If caut output format changes,
        this method is the single point to update.

        Runs every 5 minutes in background.
        Falls back gracefully: if caut not installed or returns error → empty dict.
        """

    def record_step(self, provider: str, tokens: int, cost: float) -> None:
        """Record session usage from provider output."""

    def session_summary(self) -> dict:
        """Return {total_tokens, total_cost, per_provider breakdown}."""
```

**Token extraction per provider:**
- **Claude** (`stream-json`): each JSON line may contain `"usage": {"input_tokens": N, "output_tokens": N}`. Extracted during `_run_agent()` message processing.
- **Codex** (JSON output): token counts in response metadata (if available).
- **OpenCode/Kilo**: no token data in output. Estimate using `len(captured_text) / 4`.

Token extraction happens in `_run_agent()` alongside the existing `captured_parts` logic — same loop, just also sum tokens when the field is present.

### 6.2 Session Tracking (built-in)

Runner already captures output from providers for reflexion. We add token extraction:
- Claude `stream-json` includes `usage.input_tokens` and `usage.output_tokens`
- Codex JSON includes token counts
- OpenCode/Kilo — estimate from output length (`len(text) / 4`)

Every step → Runner sends `usage_update` event → TUI updates session counters.

## 7. Persistent "Immortal" Mode

Runner never exits unless explicitly told to.

### 7.1 Mapping to existing Phase enum

The TUI displays a simplified status, but internally uses the existing `Phase` from `state.py`:

| TUI displays | Internal `Phase` | Meaning |
|-------------|-----------------|---------|
| WORKING | `Phase.RUNNING` | Executing plan |
| STUCK | `Phase.PAUSED` | Escalation Level 3, waiting for input |
| IDLE | `Phase.COMPLETED` | Plan done, waiting for next command |
| STOPPED | `Phase.FAILED` | User explicitly quit |

No new Phase values needed. `CheckpointManager._VALID_TRANSITIONS` already supports `COMPLETED → RUNNING` (restart with new plan) and `PAUSED → RUNNING` (resume after stuck).

### 7.2 State flow

```
RUNNING → (plan done) → COMPLETED/IDLE → (new plan) → RUNNING
    ↓                         ↓
  PAUSED/STUCK → (user action) → RUNNING
    ↓
  FAILED/STOPPED (only on explicit quit)
```

### 7.3 Behavior per state

In IDLE (Phase.COMPLETED), Runner:
- Keeps TUI alive, showing completion summary
- Listens for commands (new plan, quit)
- Telegram bot stays active — user can send new plan anytime
- Usage tracking continues (limits refresh)

In STUCK (Phase.PAUSED), Runner:
- Shows interactive dialog in TUI
- Sends Telegram message with inline buttons
- Waits for user command (no timeout — waits forever)
- Does NOT exit like before — stays alive

### 7.4 Plan-less startup

`tero2 go <project>` without `--plan` starts Runner in IDLE mode. Runner is initialized with `plan_file=None`. The `run()` method skips `_execute_plan()` and enters an idle loop waiting for commands. When a `new_plan` command arrives, Runner sets `self.plan_file` and calls `_execute_plan()`.

```python
# Runner changes for plan-less init:
class Runner:
    def __init__(self, project_path, plan_file=None, config=None):
        self.plan_file = plan_file  # None = start idle
        ...

    async def run(self):
        self.disk.init()
        ...
        if self.plan_file:
            await self._execute_plan(state, shutdown_event)
        # After plan completes (or no plan) — enter idle loop
        await self._idle_loop(state, shutdown_event)

    async def _idle_loop(self, state, shutdown_event):
        """Wait for commands. Exit only on 'stop' command."""
        while not shutdown_event.is_set():
            cmd = await self._wait_for_command(timeout=1.0)
            if cmd and cmd.kind == "new_plan":
                self.plan_file = self._resolve_plan(cmd.data["text"])
                state = self.checkpoint.mark_started(str(self.plan_file))
                await self._execute_plan(state, shutdown_event)
            elif cmd and cmd.kind == "stop":
                break
```

### 8.1 TUI + Runner event loop coexistence

Textual apps run their own async event loop. Runner is also async. Solution: Runner runs as a Textual `Worker` (background async task) inside the Textual app's event loop.

```python
# tero2/tui/app.py

class DashboardApp(App):
    def on_mount(self) -> None:
        """Start Runner as a background worker inside Textual's event loop."""
        self.run_worker(self._run_runner(), exclusive=True)

    async def _run_runner(self) -> None:
        """Runner coroutine — runs in Textual's async loop."""
        await self.runner.run()
```

Textual's `run_worker()` launches the coroutine as a background task in the same asyncio loop. No second event loop needed. Runner's `asyncio.sleep()`, `asyncio.Queue`, and `asyncio.timeout` all work natively.

## 10. Dependency on MVP2.5

TUI works in two modes:

- **SORA mode** (MVP2.5 implemented): shows full pipeline with all roles (scout, architect, builder, verifier, coach), slices, tasks
- **Legacy mode** (pre-MVP2.5): shows simplified view — just executor, step count, retry count. Still useful for visualization and control.

TUI does NOT require MVP2.5 to be fully implemented. It reads whatever Runner emits. More roles = richer display.

## 11. Backwards Compatibility

- `tero2 run` works exactly as before — no TUI, no event queue
- All existing tests pass without modification
- Telegram bot gains new capabilities but existing notification flow unchanged
- `textual` is a new dependency but only imported by `tero2 go`

## 12. Future Extensions

These are NOT in scope for MVP2.6, but the architecture supports them:

- **Web dashboard** — another consumer on the event bus (WebSocket → browser)
- **Multi-project** — tabs in TUI, each project its own Runner + event bus
- **Replay** — save events to JSONL, replay in TUI for debugging
- **Voice input in TUI** — microphone → STT → command queue
