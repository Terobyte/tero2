# MVP2.6 "Dashboard" — TUI + Remote Control

> **Status:** Design approved
> **Depends on:** MVP0 (Immortal Runner), MVP1 (Reflexion + Telegram), MVP2 (Stuck Detection + Escalation), MVP2.5 (Full SORA Cycle)
> **Goal:** Full-screen terminal dashboard (Textual) with live process visualization, usage tracking, hot-swap providers, and interactive stuck resolution — plus remote control via Telegram with the same capabilities.

---

## 1. Problem

MVP0-2.5 built a powerful autonomous runner with SORA pipeline, reflexion, escalation, and Telegram notifications. But there is no way to SEE what's happening or CONTROL it in real time:

- No live visualization of pipeline phases, steps, retries
- No way to change providers mid-run without editing config files
- No usage/quota visibility — you don't know when a provider is about to hit its limit
- When stuck, the runner writes files and sends Telegram text — but there's no interactive dialog
- No persistent "always-on" mode — runner exits after plan completion

## 2. Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| TUI framework | Textual | Async-native (matches Runner), CSS-like styling, built-in widgets (Log, DataTable, ProgressBar), hotkeys. Best Python TUI framework. |
| Runner ↔ TUI coupling | Event Bus (Approach A) | Runner stays independent. TUI/Telegram/nothing can attach. Runner works headless on servers. |
| Communication | asyncio.Queue (events down, commands up) | Zero external deps. Works in-process. Same mechanism for TUI and Telegram. |
| Usage: account limits | External CLI (`caut`) | Supports 16+ providers, handles cookies/OAuth. Building this ourselves = massive scope. |
| Usage: session tracking | Built-in (parse provider output) | We already capture output for reflexion. Adding token counting is trivial. |
| Layout | Panel-based (like htop/lazygit) | User chose this. One screen, all info visible, no tab switching. |

## 3. Event Bus

Runner and consumers (TUI, Telegram) communicate through a two-way "radio" system.

### 3.1 Events (Runner → consumers)

```python
# tero2/events.py

@dataclass
class Event:
    timestamp: datetime
    kind: str           # event type
    role: str           # "builder", "scout", etc. or "" for system events
    data: dict          # event-specific payload
    priority: bool = False  # True = never dropped by EventDispatcher

# Event kinds:
#   "phase_change"   — SORA phase transition (data: phase, slice, task)          priority=True
#   "step"           — step within a task (data: step, max, action)               priority=False
#   "stuck"          — runner is stuck (data: signal, level, options)             priority=True
#   "done"           — plan completed (data: tasks_completed, time_elapsed)       priority=True
#   "error"          — non-recoverable error (data: message)                      priority=True
#   "provider_switch"— provider changed (data: role, old, new)                   priority=True
#   "usage_update"   — usage data refreshed (data: limits, session_tokens, cost) priority=False
#   "escalation"     — escalation level changed (data: level, action)            priority=True
#   "log"            — general log message (data: message, severity)              priority=False
```

### 3.2 Commands (consumers → Runner)

```python
@dataclass
class Command:
    kind: str           # command type
    data: dict          # command-specific payload
    source: str = ""    # "tui", "telegram", "system" — for logging/debugging

# Command kinds:
#   "switch_provider" — data: {"role": "builder", "provider": "claude"}
#   "skip_task"       — data: {} (skip current task, move to next)
#   "steer"           — data: {"text": "try using async instead"}
#   "pause"           — data: {}
#   "resume"          — data: {}
#   "stop"            — data: {}
#   "retry"           — data: {} (retry current task from scratch)
#   "new_plan"        — data: {"text": "plan in docs/plan.md"} (free text — Runner tries
#                       as file path first, falls back to Coach interpretation if not a valid path)
```

### 3.3 EventDispatcher

A single `asyncio.Queue` only delivers to one consumer (`.get()` removes the item). TUI and Telegram both need every event. Solution: `EventDispatcher` — Runner pushes events to it, dispatcher copies to all subscribers.

```python
# tero2/events.py

class EventDispatcher:
    """Fan-out dispatcher. Runner calls emit(), consumers subscribe().

    Each subscriber gets its own asyncio.Queue. Events are copied to all.
    """

    def __init__(self) -> None:
        self._subscribers: list[asyncio.Queue[Event]] = []

    def subscribe(self) -> asyncio.Queue[Event]:
        """Create and return a new queue for a consumer.

        maxsize=500 — if a consumer falls behind, oldest events are dropped
        via put_nowait (the consumer sees current state, not a stale backlog).
        """
        q: asyncio.Queue[Event] = asyncio.Queue(maxsize=500)
        self._subscribers.append(q)
        return q

    def unsubscribe(self, q: asyncio.Queue[Event]) -> None:
        self._subscribers.remove(q)

    async def emit(self, event: Event) -> None:
        """Copy event to all subscriber queues.

        Priority events (done, error, stuck, escalation, phase_change, provider_switch)
        are never dropped. If the queue is full and the event is priority, the oldest
        NON-priority event is discarded to make room. If all 500 items are priority
        events, the queue grows by one (unbounded in that extreme case).

        Non-priority events (step, log, usage_update): oldest event discarded if full.
        Never blocks the Runner.
        """
        for q in self._subscribers:
            if q.full():
                if event.priority:
                    # Find and drop oldest non-priority item to protect critical events
                    items = []
                    dropped = False
                    while not q.empty():
                        try:
                            item = q.get_nowait()
                            if not dropped and not item.priority:
                                dropped = True  # discard this one
                            else:
                                items.append(item)
                        except asyncio.QueueEmpty:
                            break
                    for item in items:
                        q.put_nowait(item)
                    if not dropped:
                        # All 500 items are priority — append directly to the internal
                        # deque to bypass the maxsize check (one-item intentional overflow).
                        # Bounded in practice: non-priority events (step, log) normally
                        # dominate, so true all-priority fills are rare.
                        q._queue.append(event)  # type: ignore[attr-defined]
                        continue  # skip the put_nowait below
                else:
                    try:
                        q.get_nowait()  # discard oldest non-priority
                    except asyncio.QueueEmpty:
                        pass
            q.put_nowait(event)
```

**Setup:**
```python
dispatcher = EventDispatcher()
command_queue: asyncio.Queue[Command] = asyncio.Queue()

# TUI subscribes
tui_events = dispatcher.subscribe()

# Telegram subscribes
tg_events = dispatcher.subscribe()

# Runner emits — both consumers get the event
await dispatcher.emit(Event(...))
```

Runner receives `dispatcher: EventDispatcher | None` and `command_queue: asyncio.Queue[Command] | None` at init. If dispatcher is None — headless mode, no events emitted. Runner checks `command_queue` at phase boundaries (between tasks, between slices, after stuck detection). Commands from TUI and Telegram go into the same `command_queue`.

**Command ordering:** Commands are processed FIFO. Stale commands (e.g., "resume" when not paused) are silently ignored — Runner checks preconditions before acting.

**Command timing:** The command queue is checked at task boundaries (between tasks, between slices, after stuck detection). A `pause` or `stop` command sent mid-task takes effect only after the current task finishes or fails. For immediate stop, use SIGTERM/SIGINT — these set `shutdown_event` and are checked mid-stream inside `run_agent()`.

**TUI pending state:** When TUI sends a `stop` or `pause` command and the runner is mid-task, the Pipeline panel immediately shows a pending indicator ("Stopping…" or "Pausing…") to make clear the command was received but not yet effective. The indicator clears when the runner actually transitions phase.

## 5. Hot-Swap Providers

### 5.1 TUI Flow

1. Press `[r]` → modal popup with all roles and current providers
2. Select a role → second popup with available providers (claude, codex, opencode, kilo)
3. Select provider → popup closes, TUI sends `Command(kind="switch_provider", data={"role": "builder", "provider": "claude"})`
4. Runner receives command at next boundary → updates `config.roles[role].provider` in memory → next step uses new provider
5. Pipeline panel immediately reflects the change

**Known limitation — swap is in-memory only:** The provider change is NOT written to `config.toml`. After a crash or restart, the runner re-reads disk config and the original provider is restored. To make the change permanent, edit `config.toml` manually. (Persisting hot-swaps to disk is out of scope for MVP2.6.)

```
┌─ Roles ──────────────────────────┐
│                                   │
│  scout:     kilo     [изменить]  │
│  architect: claude   [изменить]  │
│  builder:   opencode [изменить]  │
│  verifier:  codex    [изменить]  │
│  coach:     claude   [изменить]  │
│                                   │
│  [Esc] закрыть                   │
└───────────────────────────────────┘
```

### 5.2 Telegram Flow

Same capability, text-based. Commands use `/` prefix to distinguish from plan text:
- `/builder claude` → switches builder to claude
- `/scout kilo` → switches scout to kilo
- `/status` → current roles + phases + usage
- `/skip` → skip current task
- `/pause` → pause runner
- `/resume` → resume after pause
- `/stop` → stop runner
- `/steer try async instead` → inject steer text

Messages without `/` prefix are treated as plan text (existing behavior).

**Integrated mode command → Command mapping** (`telegram_input.py` parses inbound messages):
```python
# Pattern matching in integrated mode:
"/builder <provider>"  → Command(kind="switch_provider", data={"role":"builder","provider":"<provider>"}, source="telegram")
"/scout <provider>"    → Command(kind="switch_provider", data={"role":"scout","provider":"<provider>"}, source="telegram")
"/skip"                → Command(kind="skip_task", data={}, source="telegram")
"/pause"               → Command(kind="pause", data={}, source="telegram")
"/resume"              → Command(kind="resume", data={}, source="telegram")
"/stop"                → Command(kind="stop", data={}, source="telegram")
"/status"              → handled locally — read EventDispatcher state, no Command emitted

<anything else>        → Command(kind="new_plan", data={"text":"<message>"}, source="telegram")
```

### 5.3 Telegram Inline Keyboard (Stuck Mode)

When Runner emits a `stuck` event in integrated mode, `telegram_input.py` sends a message with an inline keyboard:

```

[1. Сменить провайдер] [2. Скипнуть задачу]
```

**Implementation:**
- Use `reply_markup=InlineKeyboardMarkup` with `callback_data` matching command kinds
- Button 1: callback `"stuck_swap"` → **if TUI is active**: TUI opens the provider selector popup (same as pressing `[r]`). **If running headless** (no TUI): Telegram replies "Отправь: /builder <provider>" and waits for the next message.
- Button 2: callback `"stuck_skip"` → `Command(kind="skip_task", ...)`

On callback, answer the query (to dismiss the loading spinner) then put the corresponding Command into `command_queue`. ## 6. Usage Tracking

### 6.1 Account Limits (via `caut`)

New module: `tero2/usage_tracker.py`

```python
class UsageTracker:
    async def fetch_limits(self) -> dict[str, float]:
        """Call `caut usage --json`, return {provider: percent_remaining}.

        Expected caut output (JSON array of provider objects):
        [{"provider": "claude", "usage_pct": 22.0, "remaining_pct": 78.0, ...}, ...]

        We extract `remaining_pct` per provider. If caut output format changes,
        this method is the single point to update.

        Runs every 5 minutes in background (staggered by random 0-30s offset at startup
        to avoid multiple instances hammering caut simultaneously).
        Falls back gracefully: if caut not installed or returns error → empty dict.

        Validates output against expected schema: list of dicts with "provider"
        and "remaining_pct" keys. On schema mismatch, logs a warning and returns
        empty dict — never crashes on unexpected caut output format.

        Tested against caut v0.x (JSON output). If caut is not installed or the
        version does not support `--json`, the fallback returns empty dict and
        limit bars are hidden in TUI.
        """

    def record_step(self, provider: str, tokens: int, cost: float, is_estimated: bool = False) -> None:
        """Record session usage from provider output.

        is_estimated=True when token count is derived from len(text)/4 rather
        than parsed from provider output. Propagated to usage_update events so
        TUI can show '~45.2K' (estimated) vs '45.2K' (exact).
        """

    def session_summary(self) -> dict:
        """Return {total_tokens, total_cost, per_provider breakdown, is_estimated}."""
```

**`usage_update` event data includes `is_estimated: bool`** — TUI shows `~45.2K tok` prefix when estimated.

**Token extraction per provider:**
- **Claude** (`stream-json`): each JSON line may contain `"usage": {"input_tokens": N, "output_tokens": N}`. Extracted during `_run_agent()` message processing.
- **Codex** (JSON output): token counts in response metadata (if available).
- **OpenCode/Kilo**: no token data in output. Estimate using `len(captured_text) / 4`.

**Extraction is best-effort:** If expected fields are absent (provider updated its output format), fall back to `len(text) / 4` estimate and log a one-time warning. Never crash on missing token data.

Token extraction happens in `_run_agent()` alongside the existing `captured_parts` logic — same loop, just also sum tokens when the field is present.

### 6.2 Session Tracking (built-in)

Runner already captures output from providers for reflexion. We add token extraction:
- Claude `stream-json` includes `usage.input_tokens` and `usage.output_tokens`
- Codex JSON includes token counts
- OpenCode/Kilo — estimate from output length (`len(text) / 4`)

Every step → Runner sends `usage_update` event → TUI updates session counters.

## 7. Persistent "Immortal" Mode

Runner never exits unless explicitly told to.

### 7.1 Mapping to existing Phase enum

The status bar uses `Phase` for high-level runner state. The Pipeline panel uses `SoraPhase` for per-role detail (received via `phase_change` events).

**Status bar — `Phase` mapping:**

| TUI displays | Internal `Phase` | Meaning |
|-------------|-----------------|---------|
| WORKING | `Phase.RUNNING` | Executing plan |
| STUCK | `Phase.PAUSED` | Escalation Level 3, waiting for input |
| IDLE | `Phase.COMPLETED` | Plan done, waiting for next command |
| STOPPED | `Phase.FAILED` | User explicitly quit |

**Pipeline panel — `SoraPhase` mapping** (from MVP 2.5 `SoraPhase` enum):

| Pipeline shows | `SoraPhase` | Meaning |
|---------------|------------|---------|
| Hardening... | `HARDENING` | Plan review + fix loop |
| Scout running | `SCOUT` | Codebase reconnaissance |
| Coach thinking | `COACH` | Strategy / TASK_QUEUE update |
| Architect planning | `ARCHITECT` | Slice plan generation |
| Building S01/T3 | `EXECUTE` | Builder → Verifier per task |
| Slice complete | `SLICE_DONE` | Between slices |
| (legacy mode) | `NONE` | Single executor, no SORA roles |

No new Phase values needed. `CheckpointManager._VALID_TRANSITIONS` already supports `COMPLETED → RUNNING` (restart with new plan) and `PAUSED → RUNNING` (resume after stuck).

### 7.2 State flow

```
RUNNING → (plan done) → COMPLETED/IDLE → (new plan) → RUNNING
    ↓                         ↓
  PAUSED/STUCK → (user action) → RUNNING
    ↓
  FAILED/STOPPED (only on explicit quit)
```

### 7.3 Behavior per state

In IDLE (Phase.COMPLETED), Runner:
- Keeps TUI alive, showing completion summary
- Listens for commands (new plan, quit)
- Telegram bot stays active — user can send new plan anytime
- Usage tracking continues (limits refresh)

In STUCK (Phase.PAUSED), Runner:
- Shows interactive dialog in TUI
- Sends Telegram message with inline keyboard (see §5.3 below)
- Waits for user command (no timeout — waits forever)
- Does NOT exit like before — stays alive

### 7.4 Plan-less startup

`tero2 go <project>` without `--plan` starts Runner in IDLE mode. Runner is initialized with `plan_file=None`. The `run()` method skips `_execute_plan()` and enters an idle loop waiting for commands. When a `new_plan` command arrives, Runner sets `self.plan_file` and calls `_execute_plan()`.

```python
# Runner changes for plan-less init and event bus:
class Runner:
    def __init__(
        self,
        project_path: Path,
        plan_file: Path | None = None,
        config: Config | None = None,
        dispatcher: EventDispatcher | None = None,
        command_queue: asyncio.Queue[Command] | None = None,
    ):
        self.plan_file = plan_file          # None = start idle
        self.dispatcher = dispatcher        # None = headless
        self.command_queue = command_queue  # None = no remote control
        ...

    async def run(self):
        self.disk.init()
        ...
        if self.plan_file:
            await self._execute_plan(state, shutdown_event)
        # After plan completes (or no plan) — enter idle loop
        await self._idle_loop(state, shutdown_event)

    async def _idle_loop(self, state, shutdown_event):
        """Wait for commands. Exit on 'stop' command or idle_timeout."""
        idle_timeout_s = getattr(self.config, "idle_timeout_s", 0)  # 0 = never
        idle_start = asyncio.get_running_loop().time()
        while not shutdown_event.is_set():
            cmd = await self._wait_for_command(timeout=5.0)
            if cmd and cmd.kind == "new_plan":
                self.plan_file = self._resolve_plan(cmd.data["text"])
                state = self.checkpoint.mark_started(str(self.plan_file))
                idle_start = asyncio.get_running_loop().time()  # reset idle timer
                await self._execute_plan(state, shutdown_event)
            elif cmd and cmd.kind == "stop":
                break
            elif idle_timeout_s > 0:
                elapsed = asyncio.get_running_loop().time() - idle_start
                if elapsed >= idle_timeout_s:
                    log.info(f"idle timeout ({idle_timeout_s}s) reached — exiting")
                    break

    async def _wait_for_command(self, timeout: float = 5.0) -> Command | None:
        """Poll command_queue with timeout. Returns None on timeout."""
        if self.command_queue is None:
            await asyncio.sleep(timeout)
            return None
        try:
            return await asyncio.wait_for(self.command_queue.get(), timeout=timeout)
        except TimeoutError:
            return None

    def _resolve_plan(self, text: str) -> Path:
        """Resolve plan from free text — file path or literal content.

        Try as file path first (absolute or relative to project_path).
        If valid file → return its path.
        If not a file → write text to .sora/milestones/M001/ROADMAP.md (creating
        parent dirs if needed) and return that path. Writing to ROADMAP.md (not
        INLINE_PLAN.md) ensures Plan Hardening in SORA mode finds the plan at
        the expected location.
        Runner then reads it normally via disk.read_plan().

        **Backup:** if ROADMAP.md already exists with content, it is backed up to
        ROADMAP.md.bak before overwriting. Only one backup is kept (previous .bak
        is overwritten). This is a silent operation — no user prompt.
        """
```

### 8.1 TUI + Runner event loop coexistence

Textual apps run their own async event loop. Runner is also async. Solution: Runner runs as a Textual `Worker` (background async task) inside the Textual app's event loop.

```python
# tero2/tui/app.py

class DashboardApp(App):
    def on_mount(self) -> None:
        """Start Runner as a background worker inside Textual's event loop."""
        self.run_worker(self._run_runner, exclusive=True)  # pass function ref, not coroutine instance

    async def _run_runner(self) -> None:
        """Runner coroutine — runs in Textual's async loop."""
        await self.runner.run()

    def on_worker_state_changed(self, event: Worker.StateChanged) -> None:
        """Handle Runner worker exit (crash or clean stop)."""
        if event.state == WorkerState.ERROR:
            self.notify("Runner crashed — check logs", severity="error")
        elif event.state == WorkerState.SUCCESS:
            self.notify("Runner finished")
    # Worker and WorkerState imported from textual.worker
```

Textual's `run_worker()` launches the coroutine as a background task in the same asyncio loop. No second event loop needed. Runner's `asyncio.sleep()`, `asyncio.Queue`, and `asyncio.timeout` all work natively.

## 9. Parallel Instance Safety

Multiple `tero2 go` windows (different projects or same project) must coexist without corruption.

### 9.1 Project-Level Lock

**Wraps `tero2/lock.py`:** MVP 2.6 introduces `tero2/project_lock.py` which wraps the existing `FileLock` class (from `lock.py`) in a context manager interface. The lock file path stays the same (`.sora/runtime/auto.lock`) so both `tero2 run` and `tero2 go` share the same lock. `lock.py` is kept (not deleted); `project_lock.py` wraps its implementation.

**Why not rewrite:** The existing `FileLock` already handles: `os.ftruncate` before writing PID (no race on write), `_pid_alive()` check before raising `LockHeldError` (stale lock detection), retry on stale lock. Any rewrite must preserve all three.

```python
from pathlib import Path
from contextlib import contextmanager
from tero2.errors import LockHeldError

class ProjectLockError(RuntimeError):
    """Raised when another Runner already holds the project lock."""

@contextmanager
def project_lock(sora_dir: Path):
    """Acquire exclusive flock on .sora/runtime/auto.lock.

    Wraps FileLock (from the existing lock.py implementation) with error translation:
    catches LockHeldError and re-raises as ProjectLockError with a user-friendly message.
    FileLock already has __enter__/__exit__ — this wrapper adds semantic error handling,
    not context manager interface.

    Only one Runner per project — shared by `tero2 run` and `tero2 go`.
    """
    from tero2.lock import FileLock   # import here; lock.py is kept until fully replaced
    lock = FileLock(sora_dir / "runtime" / "auto.lock")
    try:
        lock.acquire()
    except LockHeldError as exc:
        raise ProjectLockError(
            f"project already in use (PID {exc.pid}). "
            f"Kill it or use a different terminal for a different project."
        ) from exc
    try:
        yield
    finally:
        lock.release()
```

**Migration:** `lock.py` is NOT deleted in MVP 2.6 — it is kept as the implementation backing `project_lock.py`. When all callers migrate to `project_lock()`, `lock.py` can be inlined into `project_lock.py` in a future cleanup. `LockHeldError` stays in `tero2/errors.py`.

**Where the lock is acquired:** `Runner.run()` wraps its entire execution in `with project_lock(self.disk.sora_dir):`. The lock is held for the lifetime of the Runner process — released on exit, crash, or SIGTERM (OS releases flock on process death automatically).

### 9.2 Multiple projects = no problem

Each project has its own `.sora/runtime/LOCK`. Two terminals running different projects never conflict:
```
Terminal 1: tero2 go ~/projects/auth     →  locks ~/projects/auth/.sora/runtime/auto.lock
Terminal 2: tero2 go ~/projects/api      →  locks ~/projects/api/.sora/runtime/auto.lock
Terminal 3: tero2 go ~/projects/auth     →  ProjectLockError — already locked by Terminal 1
```

### 9.3 Shared external resources

**`caut usage --json`** may be called by multiple instances simultaneously. Mitigation:
- Each instance staggers its `caut` refresh interval by a random 0-30s offset at startup
- `caut` calls use a 10s timeout; if it hangs, skip silently (usage bars show "—")
- `caut` is read-only — concurrent calls are safe, just wasteful

### 9.4 Stale lock detection

On startup, if `auto.lock` file exists but `fcntl.flock()` succeeds → previous process died without cleanup. Runner logs a warning ("recovered stale lock") and continues. The `LOCK_NB` (non-blocking) flag means: if flock succeeds, the lock was stale; if it raises `BlockingIOError`, another process truly holds it.

## 10. Responsive Layout

TUI adapts to any terminal size. Smaller window = fewer controls, but critical info always visible.

### 10.1 Breakpoints

**Width breakpoints:**

| Name | Width | Layout |
|------|-------|--------|
| Full | ≥ 100 cols | Two columns: left (Pipeline + Usage + Controls), right (Log) |
| Medium | 60–99 cols | Two columns but Usage panel collapses to single-line summary |
| Compact | 40–59 cols | Single column: Pipeline → Log stacked vertically. Controls = single row of key hints |
| Minimum | < 40 cols | Pipeline header + Log only. No Controls panel. Hotkeys still work. |

**Height breakpoints:**

| Name | Height | Behavior |
|------|--------|----------|
| Full | ≥ 30 rows | All panels visible |
| Medium | 20–29 rows | Usage panel hidden, Controls inline with Pipeline |
| Compact | < 20 rows | Pipeline (2 lines) + Log fills the rest |

### 10.2 What gets cut (priority order)

When space shrinks, elements are removed in this order (first to go → last):

1. **Controls panel** — hotkeys still work, just no visible buttons
2. **Usage bars** — collapse to one-line text: "Claude 78% · Codex 100% · Session 45K"
3. **Usage entirely** — hidden, data accessible via hotkey `[u]`
4. **Pipeline role details** — collapse to: "EXECUTE S01/T3 ● step 4/15"
5. **Footer** — removed
6. **Log panel** — **never removed** (last 5 lines minimum)
7. **Pipeline phase** — **never removed** (always shows current phase)

### 10.3 Implementation

Textual CSS + `on_resize()` event handler:

```python
# tero2/tui/app.py

class DashboardApp(App):
    def on_resize(self, event: events.Resize) -> None:
        w, h = event.size
        # Evaluate combined size — check height constraints first, then width.
        # A wide-but-short terminal (e.g. 120×15) should not show full layout.
        enough_height = h >= 30  # matches §10.1 table: Full = ≥30 rows; 20-29 = Medium (Usage hidden)
        enough_width_for_two_col = w >= 60
        enough_width_for_usage = w >= 60 and enough_height
        enough_width_for_controls = w >= 40 and enough_height

        self.query_one(UsagePanel).display = enough_width_for_usage
        self.query_one(ControlsPanel).display = enough_width_for_controls
        self.query_one(UsagePanel).compact = 60 <= w < 100

        # Layout direction: only two-column when wide AND tall enough
        if enough_width_for_two_col and enough_height:
            self.query_one("#main-container").styles.layout = "horizontal"
        else:
            self.query_one("#main-container").styles.layout = "vertical"
```

Textual CSS (`styles.tcss`) uses `fr` units for flexible proportions:
```css
#left-panel { width: 1fr; min-width: 28; max-width: 40; }
#right-panel { width: 2fr; min-width: 30; }
```

## 12. Escalation Integration in SORA Mode

In legacy mode, escalation lives in the `_execute_plan` retry loop (calls `decide_escalation` / `execute_escalation` from `escalation.py`). In SORA mode, execution is inside `execute_phase` which calls Builder/Verifier. `execute_phase` must integrate with the same escalation system — not replicate it.

**How it works:**

`execute_phase` imports and calls `decide_escalation()` / `execute_escalation()` directly, the same as the legacy runner. The escalation state (`escalation_level`, `div_steps`, `escalation_history`) lives on `RunnerContext` (not on the phase handler):

```python
# Add to RunnerContext (MVP2.5 §7.1):
@dataclass
class RunnerContext:
    ...
    escalation_level: EscalationLevel = EscalationLevel.NONE
    div_steps: int = 0
    escalation_history: list[EscalationLevel] = field(default_factory=list)
```

**Cross-spec note:** These three fields are defined on `RunnerContext` in MVP2.5 §7.1 (`escalation_level`, `div_steps`, `escalation_history`). They are listed here for completeness — `execute_phase.py` uses them directly from `ctx`.

`execute_phase` checks for stuck signals after each Builder failure:
```python
if builder_failed and attempt > 0:
    stuck = check_stuck(ctx.state, ctx.config.stuck_detection)
    if stuck.signal != StuckSignal.NONE:
        action = decide_escalation(stuck, ctx.escalation_level, ctx.div_steps, ctx.config.escalation)
        ctx.state = await execute_escalation(action, ctx.state, ctx.disk, ctx.notifier, ctx.checkpoint, ...)
        ctx.escalation_level = action.level
        if action.should_pause:
            # Level 3 STUCK — fire CoachTrigger.STUCK via caller
            return PhaseResult(success=False, error="stuck:level3", data={"action": action})
```

`_execute_sora` checks the return of `run_execute`: if `result.error == "stuck:level3"`, it calls `run_coach(ctx, CoachTrigger.STUCK)` and then pauses (emits `stuck` event).

**Escalation state reset between slices:** `ctx.escalation_level` and `ctx.div_steps` are reset to defaults at the start of each new slice (same as legacy mode resetting between plans).

## 13. Dependency on MVP2.5

TUI works in two modes:

- **SORA mode** (MVP2.5 implemented): shows full pipeline with all roles (scout, architect, builder, verifier, coach), slices, tasks
